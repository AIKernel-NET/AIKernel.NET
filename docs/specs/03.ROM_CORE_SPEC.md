# 03. ROM Core Spec

## 1. Purpose
Define minimum grammar, canonicalization, and validation contracts for Relation-Oriented Markdown (ROM).

## 2. Scope
- YAML front matter (entity identity)
- Headings (scope)
- Bullets (facts/properties)
- `[[id]]` (relations)

## 3. Normative Requirements
- `RCS-001` A ROM document MUST include `entity.id` and `entity.type`.
- `RCS-002` `[[id]]` references MUST be resolvable.
- `RCS-003` Facts MUST be declared in bullet form.
- `RCS-004` A Semantic Hash MUST be computable from canonicalized content.

## 4. Canonicalization (Detailed Rules)
To guarantee hash invariance, canonicalization is applied in this order.

1. Linguistic Normalization  
   Normalize full-width/half-width characters, identifier case rules, and redundant particle noise.
2. Structural Sorting  
   Bullet sets under the same heading level are string-sorted before comparison and hashing.
3. Reference Anchoring  
   Resolve `[[entity.id]]` into fully qualified identifiers (for example `namespace.project.id`) before hashing.

Notes:
- Heading numbering variance is treated as equivalent.
- Whitespace/newline/indentation formatting noise is ignored.

## 5. Fail-Closed
- `RCS-F001` Missing mandatory IDs invalidate ROM.
- `RCS-F002` Unresolved links reject reasoning input.
- `RCS-F003` Hash mismatch is treated as signature verification failure.
- `RCS-F004` Type Consistency: Reject when `entity.type` conflicts with declared properties (including missing required properties).
- `RCS-F005` Circular Reference Guard: `IPromptValidator` MUST halt when circular references can trigger unbounded reasoning loops.

## 6. Relation Syntax
ROM relations are expressed in the following three forms.

### 6.1 Property Relation
Defines static attributes for an entity.  
Form: `* [property_name]: [[target_id]]` or `value`  
Example: `* [author]: [[user.tky]]`

### 6.2 Action Relation
Defines dynamic interaction between entities.  
Form: `* [verb] -> [[target_id]]`  
Example: `* [executes] -> [[task.validation]]`

### 6.3 Quantitative Relation
Defines numeric relations used in models such as `ModelCapacityVector`.  
Form: `* [metric_name]: {numerical_value}`  
Example: `* [reasoning_score]: 0.95`

## 7. Integrity Verification
`IPromptVerifier` validates ROM in the following sequence.

1. Schema Check  
   Confirm required YAML fields (`id`, `type`, `version`).
2. Graph Linkage Check  
   Resolve all `[[id]]` via `ISignatureTrustStore` and `IPromptRepository` against trusted nodes.
3. Semantic Hash Match  
   Verify canonicalized hash equals signed `hash` field.

## 8. Acceptance Criteria
- AC1: Equivalent formatting variants yield identical Semantic Hash.
- AC2: Documents with orphan links are rejected.
