# 01. Execution Pipeline Spec

## 1. Purpose
`Structure -> Generation -> Polish` のフェーズ境界と実行契約を定義する。

## 2. Scope
- `IThoughtProcess`
- `IExecutionOutput`
- `IOutputPolisher`
- `IModelVectorRouter`
- `IAttentionGuard`
- `IPolisherValidator`
- `IPipelineOrchestrator`
- `ExecutionResult`
- `IQueryAugmentor`
- `IQueryDecomposer`
- `IQueryRouter`
- `QueryPart`

## 3. Normative Requirements
- `EPS-001` `IThoughtProcess` は構造化成果物のみを出力しなければならない（MUST）。
- `EPS-002` `IExecutionOutput` は新規推論を追加してはならない（MUST NOT）。
- `EPS-003` `IOutputPolisher` は事実・結論を改変してはならない（MUST NOT）。
- `EPS-004` フェーズ順序は固定であり、逆流は不可（MUST）。
- `EPS-005` 各フェーズ受け渡しデータはハッシュで完全性検証する（MUST）。
- `EPS-006` 状態の不変性（State Immutability）: 各フェーズ出力は Commit 後に後続フェーズで上書きしてはならない。変更が必要な場合は Replay Dump を起点とした別ブランチ再実行を行う（MUST）。
- `EPS-007` 3 層バッファとのマッピング（Buffer Mapping）:  
  Structure は `Orchestration` / `Material` から `Thought` を抽出し、Generation は `Thought` から `Expression` を生成し、Polish は `Expression` を整形して最終 `IExecutionOutput` を確定する（MUST）。
- `EPS-008` 決定論的スケジューリング（Deterministic Scheduling）: 各フェーズ開始前に `IModelVectorRouter` でモデル再評価を行い、選定根拠を `ExecutionResult` メタデータへ記録する（MUST）。
- `EPS-009` Phase 1 Query Processing: Query 補間と分解は、ROM/CacheDB の素材化および Structure 出力確定より前に完了しなければならない（MUST）。
- `EPS-010` RAG Boundary: RAG は素材供給の Provider または Pipeline 戦略であり続ける。Core の Query 抽象は知識検索を実装してはならない（MUST NOT）。

## 4. Fail-Closed
- `EPS-F001` Structure のスキーマ検証失敗時は停止する。
- `EPS-F002` フェーズ間ハッシュ不一致時は停止する。
- `EPS-F003` 必須項目欠落時は停止する。
- `EPS-F004` Information Leakage Detection: Polish フェーズで隔離コンテキストからの漏洩を `IAttentionGuard` が検知した場合、即時 Abort する。
- `EPS-F005` Logic Divergence Check: Generation が Structure の実行プランと論理乖離した場合、Polish を停止して失敗とする。
- `EPS-F006` Query Processing Failure: Query 補間・分解・embedding 変換・query routing のいずれかが一貫して完了できない場合、Phase 1 は部分的な QueryPart state を commit せずに Abort する。

## 5. Acceptance Criteria
- AC1: 不正スキーマ入力で Generation が起動しない。
- AC2: Hash 改ざん時に `Deny/Abort` が返る。
- AC3: 正常系で `ExecutionResult` に全フェーズメタデータが残る。
- AC4: 同一入力でモデル選定根拠が再現可能である。
- AC5: Polish が意味改変を試みた場合、`IPolisherValidator` により拒否される。
- AC6: 同一 input と KernelContext では QueryPart 生成が決定論的に再現できる。
- AC7: Provider routing は capability ベースの選定根拠を記録し、RAG を Core の検索実装として扱わない。

## 7. Pipeline Sequence（実行シーケンス）
`IPipelineOrchestrator` は以下の順序で原子的実行を保証する。

1. Phase Initialization  
   実行前コンテキストハッシュを確認し、`ISignatureTrustStore` 検証済みであることを確認する。
2. Phase 1 Query Processing  
   `IQueryAugmentor` で query intent を正規化・補間し、`IQueryDecomposer` で順序付き `QueryPart` に分解する。必要に応じて `IEmbeddingProvider` で意味空間へ写像し、Provider capability に基づいて `IQueryRouter` が routing する。
3. Material Context Build  
   ROM と CacheDB の素材は Provider または Pipeline 戦略を通じて解決し、Material quarantine と provenance 分離を維持する。
4. Structure Phase  
   推論モデルでタスクを `ContextFragment` 単位に分解し、思考設計図を生成する。
5. Handoff Validation  
   Structure 出力ハッシュを Generation の親ハッシュとして記録し、連鎖を形成する。
6. Generation Phase  
   設計図に従ってコンテンツを生成し、Structure の改変を禁止する。
7. Polish Phase  
   体裁/トーンを調整し、意味改変は `IPolisherValidator` で遮断する。
8. Final Commitment  
   全フェーズハッシュチェーンを含む `ExecutionResult` を生成し、ConversationStore（Git）へコミットする。
