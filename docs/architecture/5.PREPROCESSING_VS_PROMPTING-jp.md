# Preprocessing vs Prompting  
### ― 本質はプロンプトではなく前処理である

従来の AI 開発では「プロンプトの書き方」が最重要視されてきた。  
しかし AIKernel の立場は明確である：

**プロンプトは “最後の整形” にすぎず、推論精度を決めるのは前処理である。**

---

# 1. なぜ「プロンプト設計」が誤解されてきたのか

多くの研究者・実務者は以下のように誤解している：

- プロンプトの書き方が重要 → ❌  
- 例示（Few-shot）が精度を上げる → ❌  
- RAG をそのまま渡してよい → ❌  

これらはすべて **Attention 汚染理論** によって否定される。

### Few-shot 信仰の断罪  
Few-shot は「精度向上の魔法」ではなく、  
**attention を表面構造に吸わせ、推論を停止させる毒**である。

AIKernel は Few-shot を「推論破壊要因」として再定義する。

### RAG をそのまま渡すのは「構造破壊」  
生データはノイズが多く、  
推論に必要な抽象構造を attention から追い出す。

---

# 2. 本質は「何を attention に乗せるか」である

AIKernel の思想は、プロンプトの文体ではなく **attention の制御** にある。

attention に乗せるべきは：

- **目的（purpose）**  
- **制約（constraints）**  
- **抽象構造（structure）**  
- **思考パターン（reasoning pattern）**

逆に、attention から隔離すべきは：

- 例（example）  
- 文体指示  
- RAG の断片  
- 履歴  
- ノイズ  

つまり、AIKernel が扱うのは「文章」ではなく **情報構造** である。

---

# 3. 前処理（Preprocessing）が担う役割  
### ― 決定論的制御による OS 的安定性

前処理は、以下の 3 つを OS レベルで保証する。

## 3.1 情報カテゴリ分離  
（参照：1.CATEGORY_SEPARATION_PRINCIPLES.jp.md）

- purpose  
- constraints  
- structure  
- expression  
- rag_material  
- history  

これらを混在させない。

## 3.2 コンテキスト隔離  
（参照：2.CONTEXT_ISOLATION_SPEC.jp.md）

- OrchestrationContext（推論）  
- ExpressionContext（表現）  
- MaterialContext（素材）  

推論と表現を物理的に分離する。

## 3.3 RAG の構造化（＝情報の検疫 / Quarantine）

RAG の生データは **毒（ノイズ）を含む未処理素材**である。  
AIKernel はこれをそのまま脳（LLM）に入れず、必ず以下の手順で「検疫」する：

1. 取得（RAG / Provider）  
2. 正規化（不要情報の除去）  
3. 構造化（要素分解・抽象化・意味単位化）  
4. 必要部分のみを OrchestrationContext に転写  

これは LangChain の「RAG 投げっぱなし」とは根本的に異なる。

---

# 4. Prompting の限界  
### ― 非決定論的処理に依存してはならない

プロンプトは **LLM による非決定論的処理**である。

- 文体  
- 例  
- 説明  
- 比喩  

これらは ExpressionContext に属し、  
推論には一切関与しない。

### プロンプトで推論を改善することはできない  
推論の質は **前処理（決定論的処理）で決まる**。  
プロンプトは「見た目」を整えるだけである。

---

# 5. 前処理は「職人芸」ではなく「データ構造設計」である  
### ― AI 開発をスケールさせるための必須条件

プロンプトエンジニアリングはしばしば「職人の勘」に依存する。  
しかし前処理は **データ構造の設計** であり、再現性・拡張性・自動化が可能である。

AIKernel が重視するのは：

- 勘ではなく構造  
- 技術ではなく設計  
- 文章ではなくカテゴリ  
- 感覚ではなく決定論  

これにより、AI 開発は「属人芸」から脱却し、  
**OS としてスケール可能な工学的基盤**へと進化する。

---

# 6. 結論  
### ― プロンプトは「最後の整形」、推論精度は前処理で決まる

AIKernel の立場は明確である：

- プロンプトは **出力整形のための ExpressionContext**  
- 推論精度は **前処理（Preprocessing）で決まる**  
- 前処理とは **カテゴリ分離・構造化・隔離・検疫** のことである  
- つまり、AIKernel の本質は **Prompting ではなく Preprocessing** にある  

---

# 7. AIKernel における位置づけ

Preprocessing vs Prompting の思想は、AIKernel の以下の設計原則の基盤となっている：

- 情報カテゴリ分離  
- コンテキスト隔離  
- Attention 汚染理論  
- 表面模写モード防止  
- 前処理中心主義（Preprocessing over Prompting）  
- deterministic replay の保証  

AIKernel は、プロンプトの工夫ではなく **情報構造の設計** によって  
推論の純度を最大化する OS である。
