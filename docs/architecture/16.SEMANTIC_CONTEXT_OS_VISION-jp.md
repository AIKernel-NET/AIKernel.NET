# Semantic Context OS ビジョン

## 1. Summary
AIKernel.NET の最終アーキテクチャ目標は **Semantic Context OS** である。  
この OS は、AI の推論（Reasoning）・素材（Material）・生成表現（Expression）を独立したシステムオブジェクトとして統治し、推論の分岐・再利用・監査・合議を構造的に成立させる。

## 2. Background
従来の AI アプリケーション（Wrapper-based architecture）では、プロンプト、データ、履歴が単一テキストバッファに混在し、ブラックボックス化しやすい。  
その結果、次の問題が発生する。

- 再現性の欠如: 実行時コンテキスト状態が不透明で、同一結果を保証しにくい。
- 統治性の脆弱性: どの指示がどの出力へ影響したかの因果追跡が困難。
- Attention の散逸: 低価値情報（冗長履歴、装飾文体など）が推論資源を占有する。

AIKernel はコンテキストを計算資源かつ統治対象として OS レベルで制御し、モデルを交換可能な CPU として抽象化することでこれを解決する。

### 2.1. 推論純度の維持と「思考汚染」の回避
AIKernel は、推論の質を「思考の長さ（Token Count）」ではなく「構造の純度（Structural Purity）」によって定義する。

- Thinking モードの限界（The Fallacy of Depth）:  
  Thinking モード（CoT 推論）は、しばしば「深く考える」のではなく、単に「長く考える（推論ステップを線形に積み上げる）」傾向にある。この過程で生成される膨大な中間仮説や冗長な推論ステップは、内部コンテキストとして蓄積され、最終的な出力に対するノイズとなる。
- 抽象汚染（Abstract Pollution）の防止:  
  AIKernel のような高度に抽象化されたアーキテクチャ設計において、Thinking モードは「設計の自己矛盾（設計逸脱）」を引き起こしやすい。中間推論が長引くほど、初期の厳格な依存ルール（REPO_DEPENDENCY_RULES 等）が推論の重みの中で薄まり、抽象レイヤーが逆転する逆参照やレイヤー境界の侵食が発生する。
- 高速モードの戦略的活用（Fast-as-Pure 原則）:  
  明確な契約（Contracts）と構造化された素材（ROM）が提供されている場合、高速モデルによる瞬時推論の方が、中間バイアスを排除した純度の高い結果をもたらす。高速モードは、コンテキスト汚染が発生する前に結論を出すための安全な推論として再定義される。

## 3. Principles and Specification

### 3.1 Forkable Reasoning（分岐可能な推論）
AIKernel は推論途中状態を Semantic Snapshot として Markdown / JSON で保存し、Git 互換の履歴意味論で管理する。

- Snapshot 指向: Structure / Generation / Polish の境界で不変スナップショットを生成する。
- 構造的 Fork: 任意過去スナップショットから異なる `IThoughtProcess` を試行し比較できる。
- Provider 差分分析: 同一 Orchestration / Material スナップショットを異なる `IModelProvider` に投入し、モデル固有偏りを抽出できる。

### 3.2 Committee Reasoning（合議制推論）
単一モデルへ依存せず、複数独立推論系（Committee）を統合する。

- Conflict Resolution: 複数モデルの Structure 出力差分を抽出し、矛盾解消ステップを実行する。
- Evidence-based Consensus: 多数決ではなく Material 証拠整合スコアに基づいて採択する。
- Dynamic Routing 活用: 低コスト予備推論 + 高性能査読（Peer Review）を組み合わせる。

### 3.3 Signed Prompt Governance（Fail-Closed）
カーネルは署名・承認されていない Orchestration 命令を実行しない。

Fail-Closed 規則:
- 未署名または失効署名を含むパイプラインは `IPromptVerifier` により実行前遮断される。
- Signed Manifest として、プロンプト、許容ツール、予算（Token/Cost）を一体署名する。
- すべての実行は Replayable Execution Dump と紐付き、承認主体を永続証明できる。

### 3.4 Three-Layer Buffer Model（三層隔離メモリモデル）
AIKernel `MMU`（Memory Management Unit）はコンテキストを役割別に物理・論理隔離する。

- Orchestration（O-Layer）: 目的、制約、署名済み命令。
- Material（M-Layer）: 検疫済み外部証拠。
- Expression（E-Layer）: 生成結果と対話応答。

構造効果:
- 自己汚染防止: 過去 Expression が暗黙に Material へ再流入することを防ぐ。
- Attention 効率: 必要カテゴリのみを Attention に乗せ、不要層を swap-out して精度を維持する。

### 3.5. 推論モードの動的選択（Reasoning Mode Optimization）
タスクの抽象度とコンテキストの複雑性に応じ、`IModelVectorRouter` は推論モードを動的に選択する。

| モード | 特徴 | リスク | 適したタスク |
|---|---|---|---|
| Fast Mode | 瞬時変換・低レイテンシ | 複雑な論理破綻 | 構造化された DTO 変換、既存仕様の適用、単純な Polish |
| Thinking Mode | 段階的推論・高密度 | コンテキスト汚染、抽象の逆転 | 未知のアルゴリズム構築、複雑な競合解決 |

- コンテキスト蓄積による品質低下:  
  Thinking モードにおける中間推論は、モデル自身が生成した未検証の文脈である。これが蓄積されることで、本来守るべき設計原則よりも、直近の中間推論を優先する自己強化ループに陥る。これは AIKernel が最も忌避すべき Attention Pollution の一形態である。
- 推論の再帰的検疫:  
  Thinking モードを使用する場合、その中間プロセスが `Structure` フェーズを逸脱していないかを監視する必要がある。抽象レイヤーが深いタスク（OS 的アーキテクチャ設計など）では、思考の長期化が設計の純粋性を損なうため、原則として高速モードまたは「ステップを細分化した短距離推論」の連続体として処理することを推奨する。

## 4. Rationale（理論的根拠）
Semantic Context OS は次の目標を単一アーキテクチャで統合する。

- Deterministic Replay: Seed 固定、Provider/Version 固定、Material hash 固定などの条件下で再現可能性を担保する。
- Model Agnostic: 特定ベンダー名ではなく Capability 要件で実行先を選定する。
- Audit and Governance: 署名とダンプにより法的・組織的責任追跡を可能にする。
- Semantic Economy: 再利用可能スナップショットで重複推論コストを削減する。
- Pure Reasoning: カテゴリ分離により Attention 汚染を抑制し論理純度を高める。

## 5. Conclusion
AIKernel.NET は単なるプロンプトオーケストレーターではない。  
AI 推論ライフサイクル全体を統治し、コンテキスト意味論を制御する OS である。

Semantic Context OS ビジョンは、AIKernel.NET における Interface 定義、Provider 実装、将来拡張を拘束する恒久的な北極星である。

## 6. Governance / Constraints（ガバナンス・制約）
### 6.1. 推論プロセスに対するガバナンス（Process Governance）
ガバナンスの対象は、プロンプト（Input）と結果（Output）だけでなく、その中間プロセス（Reasoning Process）にも及ぶ。

- 思考ステップの制約（Step Constraint）:  
  AIKernel は、ガバナンス・ポリシーを通じて、特定の抽象レイヤー操作における Thinking ステップ数に上限を設けることができる。これは、モデルが長く考えることで設計原則を解釈し直し、勝手な実装ルールを捏造（Hallucinated Constraints）することを防ぐための Fail-Closed な防御策である。
- 抽象汚染の監査（Audit of Abstract Pollution）:  
  `Replay Dump` において、Thinking モードによる中間推論が依存ルールや責務分離を侵害した形跡がある場合、その推論結果は `TrustLevel` を大幅に減衰させる。
- モデル・アグノスティックの維持:  
  特定のモデルが持つ思考の癖（Thinking Bias）に依存した設計を排除する。高速モードで再現可能な最小限の指示と構造（ROM）こそが、AIKernel における正当な知能の証明（Proof of Intelligence）となる。
