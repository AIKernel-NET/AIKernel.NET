# コンテキスト隔離仕様

AIKernel は、LLM に渡す前に情報を **カテゴリごとに厳密に隔離する**。  
この仕様は、LLM の attention を保護し、推論能力を最大化するために設計されている。

---

# 1. 隔離対象

以下の情報は **推論コンテキストに混在させてはならない**。

- 例（example / sample）
- RAG の断片
- 文体指示
- 過去の履歴
- ノイズ情報

これらはすべて **attention 汚染の主要因**であり、推論の純度を破壊する。

---

# 2. 理論的背景（Attention 汚染）

アップロード文書より：

> 「LLM の推論能力は attention の純度に依存する。attention が表面構造に吸われると、推論は停止し、表面模写モードに落ちる。」  
> （architecture_3.ATTENTION_POLLUTION_THEORY.md より）

LLM の attention は有限であり、  
抽象構造（structure）・目的（purpose）・制約（constraints）などの  
**推論に必要な情報**に集中させる必要がある。

しかし、以下が混ざると attention が汚染される：

- 例  
- 文体  
- RAG の断片  
- 履歴  
- ノイズ  

結果として：

- 推論停止  
- 表面模写モード  
- ハルシネーション増加  
- 目的の喪失  

が発生する。

---

# 3. Failure Mode（失敗モード）

コンテキスト隔離が行われない場合、以下の failure mode が発生する。

## 3.1 表面模写モード
アップロード文書より：

> 「LLM は例を見ると表面模写モードに落ちる。これは推論能力の喪失を意味する。」  
> （architecture_5.LLM_SURFACE_MODE_FAILURE.md より）

特徴：

- 文体だけ真似る  
- 推論しない  
- 構造を理解しない  
- 目的を見失う  

## 3.2 推論停止
- structure が attention に乗らない  
- reasoning depth が浅くなる  
- chain-of-thought が崩壊する  

## 3.3 ハルシネーション増加
- RAG と context が混ざると事実性が崩壊する  

## 3.4 deterministic replay の破壊
- 履歴と質問が混ざると再現性が失われる  

---

# 4. 隔離方法（3 コンテキストモデル）

AIKernel は情報を以下の 3 つのコンテキストに分離する。

---

## 4.1 OrchestrationContext（推論用）

推論に必要な情報のみを含む：

- 目的  
- 制約  
- 抽象構造  
- 思考パターン  

**例・文体・RAG は絶対に入れない。**

---

## 4.2 ExpressionContext（表現用）  
### ― 推論後適用の厳密性

ExpressionContext は **推論が完全に終了した後** にのみ適用される。

AIKernel は「推論（Orchestration）」と「表現（Expression）」を  
単なる論理的区別ではなく、**物理的に分離された 2 ステップの処理**として実行する。

これにより：

- 推論中に文体・例・比喩が attention を汚染することを防ぐ  
- LLM が表面模写モードに落ちることを回避する  
- 出力整形が推論に影響しない  

ExpressionContext に含まれる情報：

- 文体  
- 例  
- 説明  
- 比喩  

---

## 4.3 MaterialContext（素材用）  
### ― 構造化の義務

MaterialContext に含まれる RAG の断片や外部情報は、  
**生データのまま OrchestrationContext に渡してはならない**。

AIKernel では素材は必ず以下の手順で処理される：

1. 取得（RAG / Provider）  
2. 正規化（不要情報の除去）  
3. 構造化（要素分解・抽象化・意味単位化）  
4. 必要部分のみを OrchestrationContext に転写  

これは LangChain 等の「RAG をそのまま LLM に投げる」方式とは根本的に異なる。  
構造化は **ハルシネーション抑制の中核機能**である。

---

# 5. 履歴（history）の扱い  
### ― Material と Orchestration の境界

履歴（history）は attention 汚染の主要因であり、  
AIKernel では以下のように厳密に扱う。

---

## 5.1 history が Material になる場合
- 過去の事実を参照する必要があるとき  
- 外部情報として扱うべきとき  
- 「素材」として抽象化・要約が必要なとき  

この場合、history は MaterialContext に置かれ、  
構造化してから OrchestrationContext に転写される。

---

## 5.2 history が Orchestration になる場合
- 現在の目的や制約に直接影響する場合  
- 対話の継続性を維持するための最小限の情報  

この場合でも、**未加工の履歴をそのまま渡すことは禁止**。

---

## 5.3 禁止事項
- history をそのまま OrchestrationContext に混在させること  
- 長い履歴を未加工で LLM に渡すこと  

AIKernel は history を「素材として扱うべき部分」と  
「推論に必要な最小限の文脈」に分離し、  
両者を混在させないことで deterministic replay を保証する。

---

# 6. AIKernel における実装（OS レベルでの保証）

AIKernel の Kernel / Pipeline / Context Isolation は、  
この仕様を **構造的に保証するために設計されている**。

- Kernel は 3 コンテキストを明確に分離  
- Pipeline は各ステップでコンテキストを明示  
- ProviderRouter は MaterialContext を直接渡さない  
- LlmController は ExpressionContext を推論に混ぜない  

これにより、  
**カテゴリ混在が構造的に不可能**になる。

---

# 7. 目的（Why）

コンテキスト隔離の目的は以下の通り：

- **推論の純度を最大化する**  
- **attention 汚染を防ぐ**  
- **表面模写モードを回避する**  
- **LLM の抽象推論能力を最大限引き出す**  
- **deterministic replay を保証する**  
- **OS としての一貫性を保つ**
