# Semantic Memory Management Spec
3層隔離（Orchestration / Material / Expression）を維持しながら、有限トークン資源と Attention 境界条件下で AIKernel が意味メモリを管理・最適化する方式を定義する。

## 1. Concept: Context as Managed Memory
従来の LLM 実装では、コンテキストウィンドウが単一 FIFO キューとして扱われ、重要情報が押し出される問題があった。AIKernel はこれを「セグメント化メモリ空間」として再定義し、役割別にリソース配分とパージ優先度を制御する。

## 2. Memory Units and Architecture

### 2.1 `ContextFragment`
意味メモリの最小管理単位。主属性は次の通り。

- `Payload`: 本文または構造化データ
- `Priority`: パージ耐性スコア（`0.0..1.0`）
- `Category`: `Orchestration` / `Material` / `Expression`
- `Tokens`: 実消費トークン量

### 2.2 三層バッファの役割
- `Orchestration`（O-Layer）: タスク目的、制約、署名済みルール。最優先で保護される。
- `Material`（M-Layer）: 検疫済み事実、RAG 素材。推論の証拠となる。
- `Expression`（E-Layer）: 中間生成文や過去応答。最もパージ対象になりやすい。

## 3. Token Budget Policy
`TotalBudget`（モデル最大ウィンドウ）を以下で配分する。

| 領域 | 予約種別 | 特徴 |
|---|---|---|
| `Orchestration` | `Reserved (Hard)` | 実行意思を保持する最小枠。原則パージ不可。 |
| `System/Meta` | `Safe Reserve` | 署名、ハッシュ、境界管理トークンのオーバーヘッド。 |
| `Material` | `Elastic` | タスク要件で伸縮。必要時にスワップ/要約対象。 |
| `Expression` | `Volatile` | 生成作業領域。古い断片から順次破棄。 |

## 4. Priority Under Pressure (Eviction Strategy)
`UsedTokens > TotalBudget` のとき、LIFO + Semantic Priority で解放する。

1. `Expression Purge`: 古い Expression 断片から破棄。
2. `Material Summarization`: 低優先 Material を要約へ置換し、`SourceInfo` 参照（hash pointer）を保持。
3. `Orchestration Compaction`: 命令履歴を構造圧縮。ただし `IPromptVerifier` 承認済み最新制約は保護。
4. `Halt`: 整合性を保った圧縮が不可能なら中断。

## 5. Swap and Snapshot Policy
ウィンドウ溢れ情報は Snapshot Store へスワップアウトする。

- `Checkpoint Store`: フェーズ終了時メモリ状態を保存し `Deterministic Replay` を可能化。
- `Lazy Swap-in`: `IVectorMatcher` 等が過去 Material を要求した場合のみ再ロード。

## 6. Summarization and Integrity Rules
要約時は意味的完全性を維持する。

- `Provenance Persistence`: 要約後も元 `source hash` を参照可能であること。
- `Claim Boundary`: 複数事実を混ぜて別主張へ歪めないこと。
- `No Promotion`: 要約は索引であり、原本証拠と同等信頼度で扱わないこと。

## 7. Fail-Closed Conditions (Safety)
以下が発生した場合、実行を停止し監査ログを出力する。

- `Constraint Loss`: Orchestration 必須制約がトークン不足で保持不能。
- `Provenance Gap`: 圧縮後に Material の出典情報が失われる。
- `Verification Failure`: 再構成後ハッシュが署名済みプロンプト整合性を満たさない。
