# CONTRIBUTING

---

## Introduction
AIKernel.NET is an open-source project aiming to be a **Contract Driven AI Runtime**.  
This document is a cleaned-up CONTRIBUTING guide updated for the current environment (**.NET 10 / C# 14**). It describes the minimum rules and workflows to make contributions easy to join while maintaining project quality.

---

## Key assumptions
- **Target runtime**: **.NET 10**  
- **Language**: **C# 14**  
- **Repository layout (assumed)**: `src/`, `tests/`, `samples/`, `docs/`, `tools/`. CI is expected to run build and tests automatically.

---

## Pull Request guidelines

### 1. One PR per functional area
- Keep each PR scoped to **one namespace**.  
- If changes span multiple areas, split them into separate PRs.  
- Example: do not change `AIKernel.RAG.Embeddings` and `AIKernel.Models` in the same PR.

### 2. Contract (Interface / DTO / Enum) changes require prior discussion
- **Contracts (Interface / DTO / Enum)** are central to the project.  
- Any breaking change must be proposed and discussed via an Issue before implementation.  
- **Additions** are generally acceptable; **modifications or removals** require prior discussion and agreement.

### 3. Do not submit PRs that paste AI-generated code verbatim
- PRs that simply paste code generated by ChatGPT, Copilot, Claude, or similar tools without human review are **not accepted**.  
- Examples of unacceptable PRs:
  - Generated code that does not reflect project context or design intent.  
  - Large volumes of generated code that do not follow project specifications.  
  - Comments or descriptions copied verbatim from AI output.  
- **Why**: verbatim generated code undermines the contract-driven design, reduces consistency, and harms long-term maintainability.  
- **Allowed**: using AI as an aid is fine, but you **must** apply your own understanding and judgment. Document in the PR which parts were AI-assisted and how you validated them.

### 4. Keep PRs small and reviewable
- Aim for **? 200 lines** of change where practical.  
- Break large changes into multiple, focused PRs.  
- Smaller PRs are easier to review and more likely to be merged quickly.

### 5. Documentation and XML comments are required
- **Contracts are documentation**. Provide:
  - `/// <summary>` descriptions.  
  - Parameter descriptions.  
  - Return value descriptions.  
  - Exception documentation where applicable.  
- Update `docs/` and README files when behavior or public surface changes.

### 6. Mark breaking changes explicitly
- If a PR introduces a backward-incompatible change, include **[Breaking]** in the PR title.  
- Example: use the PR title `"[Breaking] Modify RetrievalQuery structure"`.  
- Include migration guidance or compatibility layers agreed in the Issue.

### 7. Preserve APM compatibility
- AIKernel.NET aims to remain compatible with APM (Agent Package Manager) mapping.  
- Avoid changes that break mappings for `skills`, `prompts`, `instructions`, `agents`, `hooks`, `MCP servers`.  
- If a change is necessary, open an Issue with alternatives and a migration plan.

### 8. Use Issues for design discussion, PRs for implementation
- Propose and discuss specification or contract changes in an Issue.  
- Implement agreed changes in a PR.  
- Do not mix design discussion and implementation in the same PR.

## Repository layout and workflow guidance

### Recommended directory structure
- `src/` ? source code  
- `tests/` ? unit and integration tests  
- `samples/` ? example apps and usage samples  
- `docs/` ? user and developer documentation  
- `tools/` ? build scripts and utilities

### Branching and commits
- Branch naming: `feature/<short-desc>`, `fix/<short-desc>`, `chore/<short-desc>`.  
- Commit messages: concise summary; include body when needed to explain rationale and impact.  
- CI: require build and tests to pass on PRs.

### Tests and CI
- Add or update tests for all changes.  
- Tests must run with `dotnet test`.  
- Include integration or end-to-end tests for significant changes.

### Static analysis and formatting
- Follow **EditorConfig** rules.  
- Resolve **Roslyn Analyzer** warnings where practical.  
- Use `dotnet format` to format code; include format checks in CI.

---

## PR checklist
Before requesting review, ensure:
- The change is limited to **one namespace**.  
- If Contracts are changed, an Issue exists and agreement has been reached.  
- The PR title includes **[Breaking]** if the change is backward-incompatible.  
- XML comments and documentation are updated.  
- Tests are added or updated.  
- CI passes locally and on the server.  
- If AI was used, the PR description documents which parts were AI-assisted and how they were validated.

---

## Review and ownership
- Small fixes can be reviewed by any maintainer.  
- We expect **owners** to emerge per namespace; owners prioritize contract integrity and design consistency.  
- Reviewers should focus on design intent, compatibility, tests, and documentation.

---

## Community etiquette
- Small contributions are welcome.  
- Prioritize **contract integrity** over implementation details.  
- When opening an Issue, state the goal, alternatives considered, and compatibility impact.  
- Keep discussions constructive and respectful.

---

## Closing
This guide reflects the current environment (**.NET 10 / C# 14**) and an assumed repository layout. Use it as a baseline and adapt practices to the project's needs. Contributions are welcome.
