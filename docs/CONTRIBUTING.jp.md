# CONTRIBUTING

---

## はじめに
AIKernel.NET は **契約駆動（Contract Driven）な AI Runtime** を目指すオープンソースプロジェクトです。  
本ドキュメントは過去の議論やリポジトリ構成の変更を踏まえ、現状（**.NET 10 / C# 14**）に合わせて清書した CONTRIBUTING ガイドラインです。誰でも参加しやすく、品質を維持するための最低限のルールとワークフローを示します。

---

## 重要な前提
- **ターゲット環境**: **.NET 10**  
- **言語**: **C# 14**  
- **想定リポジトリ構成**: `src/`, `tests/`, `samples/`, `docs/`, `tools/` を基本とするモノレポまたはマルチプロジェクト構成を想定します。CI はビルドとテストを自動実行する設定を前提とします。

---

## Pull Request の基本方針

### 1. 1 PR は 1 機能領域に限定する
- 変更は **必ず 1 つの名前空間** に限定してください。  
- 複数領域にまたがる変更は PR を分割してください。  
- 例: `AIKernel.RAG.Embeddings` と `AIKernel.Models` を同時に変更しないこと。

### 2. 契約（インターフェース / DTO / 列挙型）の変更は Issue で事前に議論する
- **契約（インターフェース / DTO / 列挙型）** はプロジェクトの中核です。  
- 破壊的変更（Breaking Change）は必ず Issue を立てて設計と移行方針を議論してください。  
- **追加**は自由ですが、**変更・削除**は事前議論と合意が必要です。

### 3. 生成AIの出力をそのまま貼る PR は禁止する
- ChatGPT、Copilot、Claude 等で生成したコードをそのまま貼り付ける PR は受け付けません。  
- 受け付けない例: 文脈や設計意図を理解していない生成コード、仕様に沿わない大量生成コード、AI 出力をそのままのコメントや説明として残すもの。  
- **理由**: 契約駆動の一貫性と長期的な保守性を損なうためです。  
- **許容**: AI を補助ツールとして使うことは歓迎しますが、必ず **あなた自身の理解と判断** を加え、PR 説明にどの部分を AI で生成し、どのように検証したかを明記してください。

### 4. PR は小さくレビューしやすくする
- 目安として **200 行以内** を推奨します。  
- 大規模変更は機能ごとに分割して提出してください。  
- 小さな PR の方がマージされやすく、レビュー負荷が低くなります。

### 5. ドキュメントと XML コメントを必須とする
- **契約はドキュメントが本体** です。以下を必須とします:  
  - `/// <summary>` による概要説明  
  - パラメータ説明  
  - 戻り値説明  
  - 必要な場合は例外説明  
- 変更に伴うドキュメント更新を忘れないでください。`docs/` と README を整備してください。

### 6. 後方互換性を壊す場合は [Breaking] を付ける
- PR タイトルに **[Breaking]** を明記してください。  
- 例: `"[Breaking] Modify RetrievalQuery structure"`  
- Issue で合意した移行手順や互換レイヤーを PR に含めてください。

### 7. APM との整合性を意識する
- AIKernel.NET は APM（Agent Package Manager）互換の構造を保つことを目指します。  
- **skills; prompts; instructions; agents; hooks; MCP servers** とマッピング可能な設計を壊す変更は禁止します。  
- 変更が必要な場合は Issue で代替案と移行計画を提示してください。

### 8. 議論は Issue、実装は PR
- 仕様変更は Issue で議論し、実装は PR で行ってください。  
- 議論と実装を同じ PR に混ぜないでください。

## リポジトリ構成とワークフローの指針

### 推奨ディレクトリ構成
- `src/` ソースコード  
- `tests/` 単体テストと統合テスト  
- `samples/` 使用例とサンプルアプリケーション  
- `docs/` ユーザー向けと開発者向けドキュメント  
- `tools/` ビルドやスクリプト

### ブランチとコミット
- ブランチ命名: `feature/<短い説明>`, `fix/<短い説明>`, `chore/<短い説明>` を推奨します。  
- コミットメッセージ: 簡潔な要約と必要なら本文で理由と影響範囲を記載してください。  
- CI はプルリク時にビルドとテストを必須にしてください。

### テストと CI
- すべての変更は対応するテストを追加または更新してください。  
- テストは `dotnet test` で実行可能であること。  
- 重要な変更は統合テストやエンドツーエンドテストを含めてください。

### 静的解析とスタイル
- **EditorConfig** に従ってください。  
- **Roslyn Analyzer** の警告は可能な限り解消してください。  
- フォーマットは `dotnet format` で整え、CI にフォーマットチェックを入れてください。

## PR 作成時のチェックリスト
- 変更は **1 名前空間** に限定されているか。  
- 契約（インターフェース / DTO / 列挙型）変更がある場合、事前に Issue が作成され合意が得られているか。  
- PR タイトルに **[Breaking]** が必要なら付けているか。  
- XML コメントとドキュメントを更新しているか。  
- テストを追加または更新しているか。  
- CI が通ることを確認しているか。  
- 生成AIを使用した場合、どの部分に AI を使い、どのように検証したかを PR 説明に明記しているか。

---

## レビューと責任分担
- 小さな修正は誰でもレビュー可能です。  
- 名前空間ごとに **担当者** を想定しています。担当者は設計整合性と契約の保守を優先してください。  
- レビューでは設計意図、互換性、テスト、ドキュメントを重視してください。

---

## コミュニティ参加の心得
- 小さな修正から歓迎します。  
- 実装よりも **契約の整合性** を重視してください。  
- Issue で議論を始める際は目的、代替案、互換性への影響を明確にしてください。  
- 貢献者同士は建設的な議論を心がけてください。
